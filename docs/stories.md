# The Stories — Spring Music Album Catalog

User stories for the Spring Music legacy app.
Grounded in `AlbumController.java`, `InfoController.java`, `ErrorController.java`, `Album.java`,
`albums.json`, `albums.js`, `app.js`, `errors.js`, `info.js`, and all HTML templates.

---

## Notable Legacy Behaviors (pin these — do not "fix" in characterization tests)

| # | Quirk | Where | Notes |
|---|-------|-------|-------|
| LB-1 | `PUT /albums` creates, `POST /albums` updates | `AlbumController.java:29-38` | HTTP verbs are swapped from REST convention — intentional legacy behavior to document |
| LB-2 | UI never calls `PUT /albums` — both add and update go through `POST /albums` | `albums.js:saveAlbum()` | Angular `$resource.save()` always issues POST; the PUT endpoint is unreachable from the standard UI. Any new album submitted via the form lands on the POST (update) handler. |
| LB-3 | `albumId` field exists alongside `id` | `Album.java` | Two identity fields — `id` is the JPA primary key (generated by `RandomIdGenerator`); `albumId` is an extra nullable field never populated by the UI or seed data |
| LB-4 | No server-side genre filtering | `AlbumController.java:24` | `GET /albums` always returns the full collection; genre filtering is done entirely client-side in `albums.js` |
| LB-5 | `trackCount` is in the domain model but absent from the UI form and seed data | `Album.java`, `albumForm.html`, `albums.json` | The field exists on the Java entity and is returned in JSON responses, but the add/edit form has no `trackCount` input and the seed data omits it (defaults to 0) |
| LB-6 | Default sort field is `"name"` but no such field exists | `albums.js:$scope.init()` | `$scope.sortField = "name"` — the actual field is `title`; on initial load the sort is effectively undefined, albums appear in API return order |
| LB-7 | `GET /albums/{id}` exists but is never called by the UI | `AlbumController.java:42` | The single-album fetch endpoint is wired up but all client reads go through `GET /albums` (full list) |
| LB-8 | `POST /errors/fill-heap` exists on the server but is not exposed in the UI | `ErrorController.java`, `errors.js`, `errors.html` | The fill-heap endpoint is backend-only; `errors.js` only registers `kill` and `throw`; no button exists in `errors.html` |

## User Stories

### Album Catalog (Core CRUD)

| # | Role | Goal | Acceptance Criteria | Tester Verifies |
|---|------|------|--------------------|--------------------|
| US-1 | Music catalog user | Browse the full list of albums so I can see what's in the collection | Given the catalog has albums, when I load the page, then all albums are displayed with title, artist, release year, and genre | `GET /albums` returns HTTP 200 with a JSON array; each album object contains `id`, `title`, `artist`, `releaseYear`, `genre`, and `trackCount` (integer, 0 if not set); empty catalog returns `[]` not an error; the 29 seed albums are present on a fresh start |
| US-2 | Music catalog user | Filter albums by genre so I can find albums in a specific style | Given albums of multiple genres exist, when I select a genre filter, then only albums matching that genre are shown | Client-side filter reduces displayed albums to the selected genre; selecting "All" (or clearing the filter) restores the full list; genres present in seed data: **Rock**, **Pop**, **Blues** (not Jazz or Classical); filter is case-sensitive string match |
| US-3 | Catalog manager | Add a new album so the collection stays current | Given I submit a valid album (title, artist, releaseYear, genre), when the form is saved, then the album appears in the list with a system-generated ID | Clicking "add an album" opens a modal with fields for title, artist, releaseYear, and genre (no trackCount input); the OK button is disabled while any required field is empty; `releaseYear` must match `/^[1-2]\d{3}$/` (years 1000–2999) — the OK button stays disabled otherwise; on submit the client issues `POST /albums` (not PUT) with the album JSON; server returns HTTP 200 with the saved album including a non-empty `id`; the album is retrievable via `GET /albums`; album count increases by 1 |
| US-4 | Catalog manager | Edit an existing album via the modal form so I can correct mistakes | Given an album exists, when I open its edit modal, change one or more fields, and click OK, then the updated values are reflected in the list | Clicking "edit" in the album's gear dropdown opens the same `albumForm.html` modal pre-populated with existing values; same required-field and year-pattern validation applies; on submit the client issues `POST /albums` with the full album JSON including the existing `id`; server returns HTTP 200 with updated fields; `GET /albums` reflects the new values; album count does not change |
| US-5 | Catalog manager | Edit a single field on an album inline without opening a modal | Given an album is displayed in grid or list view, when I click on a field value (title, artist, releaseYear, or genre), then an inline text input appears; when I press Enter or click the checkmark, the value is saved | Clicking a field value activates the in-place editor for that field only; pressing Escape or clicking the X cancels without saving; submitting an empty value is blocked (`if ($scope.newFieldValue === "") { return false; }`); saving issues `POST /albums` via `Albums.save()`; a success or error toast appears; the updated value is immediately visible |
| US-6 | Catalog manager | Delete an album so the collection doesn't contain outdated entries | Given an album exists, when I delete it, then it no longer appears in the list | Clicking "delete" in the gear dropdown calls `DELETE /albums/{id}`; server returns HTTP 200 (void); subsequent `GET /albums` does not contain the deleted album; total album count decreases by 1 |

### View and Sort

| # | Role | Goal | Acceptance Criteria | Tester Verifies |
|---|------|------|--------------------|--------------------|
| US-7 | Music catalog user | Switch between grid and list layout so I can view albums in my preferred format | Given the albums page is loaded, when I click the grid icon, albums display as Bootstrap thumbnail cards in a multi-column grid; when I click the list icon, albums display in a striped table with columns Title, Artist, Year, Genre | Default view on load is grid; both views show the same album data; both views expose the gear dropdown (edit/delete) and in-place editing on each field |
| US-8 | Music catalog user | Sort albums by field so I can find albums more easily | Given albums are displayed, when I click a sort link (title, artist, year, genre), albums reorder by that field; when I click the arrow toggle, sort direction reverses | Sort links are: title, artist, year (maps to `releaseYear`), genre; direction toggle shows chevron-up (ascending) or chevron-down (descending); sort is applied client-side via Angular `orderBy`; on initial load sort field defaults to `"name"` which matches no field, so order is undefined (see LB-6) |

### Error Simulation

| # | Role | Goal | Acceptance Criteria | Tester Verifies |
|---|------|------|--------------------|--------------------|
| US-11 | Operator | Kill the running application instance to test platform resilience and restart behavior | Given the app is running, when I navigate to `/#/errors` and click "Kill", a request is sent to `GET /errors/kill`; the server calls `System.exit(1)` and the process terminates | `GET /errors/kill` causes the JVM to exit; the client-side `errors.js` expects a 502 response (gateway down) and shows "application backend was killed: 502"; any other response status triggers "unexpected error" message; a 200 response triggers "should have been killed but returned successfully" message |
| US-12 | Operator | Throw an unhandled exception to test error reporting and logging | Given the app is running, when I click "Throw Exception" on the errors page, a request is sent to `GET /errors/throw`; the server throws a `NullPointerException` | `GET /errors/throw` returns HTTP 500; the client-side handler expects exactly 500 and shows "An error occurred as expected: 500"; any other status triggers "unexpected error" message; a 200 response triggers "exception should have been thrown but was not" message |

## API Endpoint Inventory

| Verb | Path | Handler | Called by UI? | Notes |
|------|------|---------|---------------|-------|
| GET | `/albums` | `AlbumController.albums()` | Yes | Returns full collection; no server-side filtering |
| PUT | `/albums` | `AlbumController.add()` | No | Create — HTTP verb swapped; UI never calls this (see LB-2) |
| POST | `/albums` | `AlbumController.update()` | Yes (for both add AND update) | Update — HTTP verb swapped; UI routes all saves here |
| GET | `/albums/{id}` | `AlbumController.getById()` | No | Returns album or `null`; not called from UI (see LB-7) |
| DELETE | `/albums/{id}` | `AlbumController.deleteById()` | Yes | Void return |
| GET | `/appinfo` | `InfoController.info()` | Yes | Returns `{ profiles: [], services: [] }` |
| GET | `/service` | `InfoController.showServiceInfo()` | No | Returns CF service list; API-only |
| GET | `/errors/kill` | `ErrorController.kill()` | Yes | `System.exit(1)` |
| GET | `/errors/throw` | `ErrorController.throwException()` | Yes | Throws `NullPointerException` |
| GET | `/errors/fill-heap` | `ErrorController.fillHeap()` | No | OOM loop; not wired in UI (see LB-8) |

## Data Model — Album

| Field | Java Type | JPA | UI Form | Seed Data | Notes |
|-------|-----------|-----|---------|-----------|-------|
| `id` | `String` | `@Id`, generated by `RandomIdGenerator` | Hidden (populated by server) | Not present (auto-generated) | 40-char random string primary key |
| `title` | `String` | Column | Required text input | Present | `required` in form |
| `artist` | `String` | Column | Required text input | Present | `required` in form |
| `releaseYear` | `String` | Column | Required text input | Present | Stored as String, not int; client validates `/^[1-2]\d{3}$/` |
| `genre` | `String` | Column | Required text input | Present | Free-text; no server-side enum constraint |
| `trackCount` | `int` | Column | Absent | Absent (defaults to 0) | Exists in model and JSON responses; never set by UI |
| `albumId` | `String` | Column | Absent | Absent | Ghost field — no known purpose (see LB-3) |

## Out of Scope (for this modernization exercise)

- User authentication / access control (any user can add/edit/delete)
- Album artwork / media files
- Search by title or artist (only client-side genre filter exists)
- Pagination (full list always returned)
- Audit trail / change history
- `trackCount` UI — the field exists in the model but the modernized app is not required to add a form input for it unless explicitly scoped
- `albumId` field — not reproduced in the modernized service; it has no known purpose

## Coverage Notes

**What was checked:**
- All three `@RestController` classes: `AlbumController`, `InfoController`, `ErrorController`
- Full `Album.java` domain model including field types and JPA annotations
- `RandomIdGenerator` existence (confirms ID is server-generated, not client-supplied)
- All 29 seed albums in `albums.json` — verified actual genres (Rock, Pop, Blues; no Jazz or Classical)
- `albums.js`: both `AlbumsController` (modal add/edit/delete) and `AlbumEditorController` (in-place field editing) and the `inPlaceEdit` directive
- `app.js`: routing — only two routes: `/errors` and a catch-all for albums
- `errors.js`: only `kill` and `throw` are registered; `fill-heap` is absent
- `info.js` and `InfoController`: `/appinfo` and `/service` endpoints
- `status.js`: the shared notification service used across all controllers
- All 8 HTML templates including `albumForm.html` (confirmed: no `trackCount` field)
- `index.html`: confirmed Angular module wiring and library versions (Angular 1.2.16, Bootstrap 3.1.1)

**Key corrections from the original draft:**
- US-2: Changed genre list from "Rock, Pop, Jazz, Classical" to "Rock, Pop, Blues" (actual seed data)
- US-3: Removed `trackCount` from the form's required fields; clarified the client sends POST not PUT
- US-4: Clarified this is the modal-edit path only; split in-place editing into its own story (US-5)
- US-5 (was US-5 delete): Delete renumbered to US-6; in-place edit added as US-5
- LB-2 added: The most significant finding — the UI never invokes PUT; both create and update hit the POST endpoint
- LB-6 added: Default sort bug (`"name"` field does not exist)

**Intentionally left out:**
- `fill-heap` as a user story — it exists in Java but is not reachable from the UI and has no test coverage surface in a UI-driven characterization suite
- `/service` as a functional story — it is an infrastructure diagnostic endpoint with no UI; included in API inventory only
- `GET /albums/{id}` as a testable behavior — it is dead code from the UI's perspective; included in API inventory for completeness
- Bootstrap/Angular library versions — not behavioral requirements, captured in Coverage Notes only
